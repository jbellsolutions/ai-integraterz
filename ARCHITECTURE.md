# AI Integraterz — Architecture

## Overview

AI Integraterz is a markdown-driven, agent-based system. Each agent is a Claude Code playbook (`.md` file) that runs via a bash runner (`lib/run-agent.sh`). Agents read from structured config files, write to output directories, and update state files after each run.

The system is intentionally simple: no server required, no database required for core operation, no build step. Claude Code IS the runtime.

## Key Abstractions

### Client Config (`config/clients/<id>.json`)
Single source of truth for a client. Created by `extraction.md`, read by every downstream agent. Schema defined by `config/clients/example-client.json`.

### Agent Playbook (`agents/*.md`)
A Claude Code instruction document. Contains: mission, schedule, prerequisites, step-by-step run checklist, output paths, state file format, and safety rails. Executed by `lib/run-agent.sh`.

### Run State (`state/<agent>/last-run.json`)
Written by each agent at completion. Contains: status, run_count, last_run_at, next_actions, and agent-specific data. Read by orchestrator to monitor health.

### Templates (`templates/**/*.md`)
Markdown templates with `{{PLACEHOLDER}}` tokens. Populated by `training-builder.md` with client-specific data. Never edited directly — always through the builder agent.

### Delivery Repo (`outputs/<id>/delivery-repo/`)
The client-facing package. A standalone repo structure the client can open in Claude Code and self-install by saying "set this up for me."

## Optional Module Integration

The system is modular. Two optional repos extend it:

```
~/Desktop/
├── ai-integraterz/          ← This repo (core)
├── expert-series-v3/        ← Expert Series module (optional)
│   └── agents, templates
├── gtm-company/             ← Full GTM module (optional)
│   └── agents, triggers
└── titans-of-direct-response-mastermind-council/  ← Positioning engine
```

Enabled via `config/project.json → optional_modules.<module>.enabled: true`.

---

## Agent Communication

Agents do not call each other directly. Communication happens through:

1. **State files** — Agent A writes state; orchestrator reads it and triggers Agent B
2. **Slack notifications** — Agents post to `#ai-integraterz-ops` on completion
3. **Orchestrator routing** — The orchestrator reads all agent states and routes next steps

This keeps agents decoupled and independently runnable.

## Integration Points

| External System | Used By | MCP Tool |
|----------------|---------|---------|
| Slack | All agents (notifications) | `slack_send_message` |
| Google Calendar | `coaching-setup` | `gcal_create_event` |
| ClickUp | `coaching-setup` | `clickup_create_task` |
| Gmail | `orchestrator`, `delivery-packager` | `gmail_create_draft` |
| Supabase | Optional persistence | REST API via bash |

All integrations are optional for core operation. The system runs without any of them — agents just skip the notification steps.
